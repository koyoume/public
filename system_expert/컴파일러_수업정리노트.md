## Ch11. Static Single Assignment (SSA) Form — 변수에 번호를 매겨서 분석을 쉽게

### 1. SSA란?

SSA = 각 변수가 프로그램 텍스트에서 딱 한 번만 정의되는 형태. 같은 변수가 여러 번 대입되면 각각 다른 이름을 부여한다(a1, a2, a3...). "Static"의 의미: 소스 코드상 한 번. 루프 안이면 런타임에 여러 번 실행되지만 텍스트에서는 한 줄.

### 2. 왜 SSA가 필요한가?

일반 형태에서 "x"라는 이름만으로는 어느 정의에서 온 값인지 즉시 알 수 없다. Reaching Definitions 같은 분석이 필요하다. SSA에서는 변수 이름(x0, x1, x2)만 보면 어느 정의인지 즉시 파악 가능 → RD 분석 자체가 불필요. AE에서 KILL이 비어짐(변수 재정의가 없으므로). Def-Use 체인이 O(N*M) → O(N+M)으로 축소.

### 3. 제어 흐름 없는 경우 — 변환이 쉬움

각 정의에 새 이름, 이후 사용에 전파: a=x+y → a1=x+y, b=a-1 → b1=a1-1, a=y+b → a2=y+b1...

### 4. 제어 흐름이 있는 경우 — phi-function

분기 후 합류점에서 "어느 경로로 왔느냐에 따라 다른 변수를 선택"하는 표기법. phi(a2, a1)에서 첫 인자는 첫 predecessor 경로, 두 번째는 두 번째 경로. 실제로 실행되는 함수가 아니라 분석/최적화용 표기(notational fiction).

### 5. Dominance Frontier — phi를 어디에 삽입?

모든 합류점에 phi를 넣으면 비효율적. 필요한 곳에만 삽입한다. DF(x) = {w | x가 w의 어떤 predecessor를 dominate하면서 x가 w를 strictly-dominate하지 않는 노드}. 직관: x의 지배 영역 경계. 변수 a를 정의하는 BB x에 대해, DF(x)의 각 노드에 a의 phi를 삽입.

이름 변경(Renaming): dominator tree를 BFS로 방문하면서, 각 use를 dominator tree에서 가장 가까운 def의 이름으로 교체.

### 6. SSA 기반 최적화 — 상수 전파 예시

일반형에서 x=1 이후 x를 사용하는 곳에 1을 전파하려면 "어느 x인지" 확인이 필요. SSA에서는 x0=1이면 x0을 사용하는 모든 곳에 바로 1을 전파. 이름이 곧 정의를 특정하므로.

### 7. Back Transformation — SSA에서 일반 형태로

최적화 후 기계어 생성을 위해 되돌림. phi(x1, x2)를 각 predecessor 끝에 copy 명령어로 변환.

Lost Copy Problem: copy propagation 후 back transform하면 순서 문제 발생 가능. 예: x=1; loop: phi(x0,x2); y=x1; x=2; return y → copy prop → return x1 → back transform에서 x1이 x2에 덮어씌워짐 → 항상 2 반환(원본은 1 반환). 해결: Edge-Split SSA Form — 간선 사이에 빈 노드 삽입으로 copy 충돌 방지.

### 8. SSA는 업계 표준

LLVM 내부 IR이 SSA. DCE, CSE, LICM, 상수 전파, value numbering 모두 SSA 기반. Android Dalvik VM, Java HotSpot VM, V8 JS Engine도 SSA 사용.

## Ch8. Register Allocation — 변수를 레지스터에 배정하기

### 1. 문제의 중요성

프로그램의 의사 레지스터(수백~수천 개)를 물리 레지스터(16~32개)에 배정. 레지스터 접근은 메모리 대비 100배 이상 빠르므로 성능에 가장 직접적인 영향. 용어: Allocation(유지 결정), Assignment(구체 레지스터 결정), Spilling(메모리로 내보냄).

### 2. 간섭과 간섭 그래프

두 변수가 어떤 프로그램 지점에서 동시에 live하면 간섭 → 다른 레지스터 필요. 간섭 그래프: 노드=live range, 간선=간섭 쌍. 레지스터 할당 = 그래프 n-컬러링 (NP-complete → 휴리스틱).

### 3. Live Range (Web)

같은 변수라도 다른 시점의 정의는 다른 live range로 분리 가능. Live range = RD에서 d가 도달 AND LV에서 변수가 live인 지점 집합. 겹치는 live range는 merge. 분리하면 그래프가 성기게 → 컬러링 쉬워짐.

### 4. Chaitin's Algorithm

핵심 관찰: degree < n인 노드는 항상 컬러링 가능 (이웃이 n-1개 이하니까 남는 색 있음).

Simplify: degree < n 노드를 스택 push + 제거 (이웃 degree 감소 → 연쇄 반응). 모든 노드 제거되면 성공. Assign: 스택에서 역순 pop, 이웃과 다른 색 배정. Stuck: 모든 degree >= n → spill 필요.

### 5. Spilling

spill 비용 = 사용/정의 횟수 x 루프 중첩 가중치(c x 10^d). spill 이득 = degree (이웃이 많을수록 풀림). cost/degree 낮은 노드를 우선 spill. spill 후 load/store 삽입 → 간섭 그래프 재구축 → 다시 simplify.

### 6. Optimistic Coloring (Briggs)

Chaitin은 degree >= n이면 즉시 spill 결정(비관적). Briggs는 일단 스택에 넣고 Assign에서 색칠 시도(낙관적). 실제로 색 없으면 그때 spill. 다이아몬드 그래프(n=2)에서 Chaitin은 spill하지만 Briggs는 2색 가능. Chaitin보다 같거나 적은 spill.

### 7. Live Range Splitting

Spill 대신 live range를 쪼개서 간섭 줄이기. Case 1: 거의 죽은 구간 분리 (교대로 같은 레지스터 사용). Case 2: copy 삽입으로 다른 레지스터 배정 (spill 회피). LLVM에서 greedy allocator + aggressive splitting이 높은 성능.

### 8. Copy Coalescing

COPY x,y에서 x,y가 간섭 안 하면 합쳐서 같은 레지스터 → COPY 삭제. 단, 합치면 degree 증가 → 컬러링 어려워질 수 있음. 전략: Aggressive/Conservative/Iterated/Optimistic.

### 9. Caller-Save vs Callee-Save

Caller-save: 호출자가 저장/복원. 호출 경계 안 넘는 변수에 배정. Callee-save: 피호출자가 entry/exit에서 1번 저장/복원. 호출 경계 넘는 변수에 배정. 최적: 호출 안 넘는 변수→caller-save, 넘는 변수→callee-save. Leaf 함수는 caller-save만으로 충분.

### 10. Pre-colored 레지스터

SP, FP, GP, 함수 인자, 리턴값 등 미리 물리 레지스터 고정. 간섭 그래프에 pre-colored 노드로 삽입, copy coalescing으로 자연스럽게 처리.
