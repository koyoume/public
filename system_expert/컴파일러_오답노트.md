## Q1 -- Ch2 지역 최적화 연쇄 적용 | 부분 정답

**문제**: BB에 대해 지역 최적화를 순서대로 적용하라. (s1: t1=a+b, s2: t2=a+b, s3: t3=t2*c, s4: a=t1+t3, s5: t4=a+b)

**내 답변**: Local CSE로 s2 삭제, t2=t1로 대체 [X] (CSE까지만 답변)

**정답 및 보완**: CSE 적용 후 Copy Propagation(t2->t1 전파) -> Dead Code Elimination(t2 제거)까지 연쇄 적용해야 완전한 답. s5의 a+b는 s4에서 a가 재정의되었으므로 CSE 대상이 아님. 최적화 결과: s1: t1=a+b, s3: t3=t1*c, s4: a=t1+t3, s5: t4=a+b (5->4문장)

**핵심 개념**: 지역 최적화는 CSE -> Copy Propagation -> DCE 순서로 연쇄 적용하며, 변수 재정의 이후의 동일 표현식은 CSE 불가

## Q2 -- Ch3 Reaching Definitions GEN/KILL 계산 | 출제 오류 + 부분 정답

**문제**: CFG에서 RD의 GEN, KILL 구하고 1회 iteration 후 IN/OUT 계산 (출제 시 정의 번호 중복 오류 있었음)

**내 답변**: 풀이 방향 맞았으나 정의 번호 중복 지적 [O] (오류 발견)

**정답 및 보완**: KILL 계산 시 해당 블록에서 정의하는 변수의 "프로그램 전체의 다른 모든 정의"를 포함해야 함. 예: B1에서 x=1, y=2 정의 시 KILL[B1] = {x의 다른 정의} U {y의 다른 정의}. KILL은 정적 속성으로 IN과 무관하게 고정.

**핵심 개념**: KILL = 같은 변수를 정의하는 프로그램 전체의 다른 정의 (정적 수집, IN 무관)

## Q3 -- Ch1 Instruction Scheduling 2회 수행 이유 | 부분 정답

**문제**: Instruction Scheduling이 RA 전후 두 번 수행되는 이유와 한 번으로 부족한 이유

**내 답변**: RA 후 재최적화 가능하므로 재시도 [X] (RA 전 목적 누락)

**정답 및 보완**: RA 전 - 가상 레지스터 무한 가정 하에 ILP 최대화(파이프라인 stall 감소). RA 후 - spill code(load/store) 삽입으로 새로운 stall 발생하므로 재스케줄링 필요. 한 번으로 부족한 이유: RA 전에만 하면 spill 미반영, RA 후에만 하면 레지스터 제약으로 ILP 최대화 불가. Phase-ordering problem의 대표 사례.

**추가 심화 (Q&A에서 도출)**: RA 후 2차 스케줄링은 단순히 spill code 추가 최적화만이 아니라, RA가 만든 문제도 보정한다. (1) spill code(load/store) 삽입으로 인한 새 stall 배치 최적화 (추가 최적화). (2) 여러 가상 레지스터가 같은 물리 레지스터에 할당되면서 이전에 없던 anti dependence(read->write), output dependence(write->write)가 새로 발생 -- 기존 스케줄링 순서가 이 false dependence와 충돌할 수 있어 보정 필요. 현재 실무 접근(current wisdom): 무한 가상 레지스터 -> Schedule -> Assign 레지스터 -> Peephole Schedule (spill code 주변만 국소 재스케줄링).

**핵심 개념**: RA 전=ILP 최대화, RA 후=spill code 배치 + RA가 만든 false dependence 보정 (Phase-ordering problem). 2차 스케줄링은 "추가 최적화 + 기존 보정" 둘 다 수행

## Q4 -- Ch4 합류 연산자 Union vs Intersection | 정답

**문제**: RD가 합집합, Available Expressions가 교집합을 쓰는 이유를 보수적 근사 관점에서 설명

**내 답변**: RD는 가능한 경우의 수 모두 고려, AE는 확실한 것만 고려 [O]

**정답 및 보완**: 합집합(Union) = may 분석 (있을 수도 있는 것 모두 포함), 교집합(Intersection) = must 분석 (반드시 성립하는 것만). RD에서 빠뜨리면 살아있는 정의를 제거하는 잘못된 최적화 발생. AE에서 과다 포함하면 계산 안 된 표현식을 재사용하는 오류 발생.

**핵심 개념**: may 분석(Union) = 안전하게 과대 추정, must 분석(Intersection) = 안전하게 과소 추정

## Q5 -- Ch3 KILL 정적 속성 | 문제 서술 부족

**문제**: "KILL이 반복마다 달라진다"는 학생의 접근이 틀린 이유 설명

**내 답변**: 문제 서술이 모호하여 판단 어려움 [--]

**정답 및 보완**: KILL은 프로그램 전체에서 같은 변수를 정의하는 다른 모든 문장을 정적으로 수집한 것. IN 내용과 무관하게 고정. KILL이 실제로 "죽이는" 효과는 전달함수 OUT = GEN U (IN - KILL)에서 IN과 만날 때 발현되지만, KILL 집합 자체는 반복 과정에서 절대 변하지 않음.

**핵심 개념**: KILL 집합은 정적 속성 -- 반복 알고리즘의 IN/OUT 변화와 무관하게 사전에 고정
