## Q1 -- Ch2 지역 최적화 연쇄 적용 | 부분 정답

**문제**: BB에 대해 지역 최적화를 순서대로 적용하라. (s1: t1=a+b, s2: t2=a+b, s3: t3=t2*c, s4: a=t1+t3, s5: t4=a+b)

**내 답변**: Local CSE로 s2 삭제, t2=t1로 대체 [X] (CSE까지만 답변)

**정답 및 보완**: CSE 적용 후 Copy Propagation(t2->t1 전파) -> Dead Code Elimination(t2 제거)까지 연쇄 적용해야 완전한 답. s5의 a+b는 s4에서 a가 재정의되었으므로 CSE 대상이 아님. 최적화 결과: s1: t1=a+b, s3: t3=t1*c, s4: a=t1+t3, s5: t4=a+b (5->4문장)

**핵심 개념**: 지역 최적화는 CSE -> Copy Propagation -> DCE 순서로 연쇄 적용하며, 변수 재정의 이후의 동일 표현식은 CSE 불가

## Q2 -- Ch3 Reaching Definitions GEN/KILL 계산 | 출제 오류 + 부분 정답

**문제**: CFG에서 RD의 GEN, KILL 구하고 1회 iteration 후 IN/OUT 계산 (출제 시 정의 번호 중복 오류 있었음)

**내 답변**: 풀이 방향 맞았으나 정의 번호 중복 지적 [O] (오류 발견)

**정답 및 보완**: KILL 계산 시 해당 블록에서 정의하는 변수의 "프로그램 전체의 다른 모든 정의"를 포함해야 함. 예: B1에서 x=1, y=2 정의 시 KILL[B1] = {x의 다른 정의} U {y의 다른 정의}. KILL은 정적 속성으로 IN과 무관하게 고정.

**핵심 개념**: KILL = 같은 변수를 정의하는 프로그램 전체의 다른 정의 (정적 수집, IN 무관)

## Q3 -- Ch1 Instruction Scheduling 2회 수행 이유 | 부분 정답

**문제**: Instruction Scheduling이 RA 전후 두 번 수행되는 이유와 한 번으로 부족한 이유

**내 답변**: RA 후 재최적화 가능하므로 재시도 [X] (RA 전 목적 누락)

**정답 및 보완**: RA 전 - 가상 레지스터 무한 가정 하에 ILP 최대화(파이프라인 stall 감소). RA 후 - spill code(load/store) 삽입으로 새로운 stall 발생하므로 재스케줄링 필요. 한 번으로 부족한 이유: RA 전에만 하면 spill 미반영, RA 후에만 하면 레지스터 제약으로 ILP 최대화 불가. Phase-ordering problem의 대표 사례.

**추가 심화 (Q&A에서 도출)**: RA 후 2차 스케줄링은 단순히 spill code 추가 최적화만이 아니라, RA가 만든 문제도 보정한다. (1) spill code(load/store) 삽입으로 인한 새 stall 배치 최적화 (추가 최적화). (2) 여러 가상 레지스터가 같은 물리 레지스터에 할당되면서 이전에 없던 anti dependence(read->write), output dependence(write->write)가 새로 발생 -- 기존 스케줄링 순서가 이 false dependence와 충돌할 수 있어 보정 필요. 현재 실무 접근(current wisdom): 무한 가상 레지스터 -> Schedule -> Assign 레지스터 -> Peephole Schedule (spill code 주변만 국소 재스케줄링).

**핵심 개념**: RA 전=ILP 최대화, RA 후=spill code 배치 + RA가 만든 false dependence 보정 (Phase-ordering problem). 2차 스케줄링은 "추가 최적화 + 기존 보정" 둘 다 수행

## Q4 -- Ch4 합류 연산자 Union vs Intersection | 정답

**문제**: RD가 합집합, Available Expressions가 교집합을 쓰는 이유를 보수적 근사 관점에서 설명

**내 답변**: RD는 가능한 경우의 수 모두 고려, AE는 확실한 것만 고려 [O]

**정답 및 보완**: 합집합(Union) = may 분석 (있을 수도 있는 것 모두 포함), 교집합(Intersection) = must 분석 (반드시 성립하는 것만). RD에서 빠뜨리면 살아있는 정의를 제거하는 잘못된 최적화 발생. AE에서 과다 포함하면 계산 안 된 표현식을 재사용하는 오류 발생.

**핵심 개념**: may 분석(Union) = 안전하게 과대 추정, must 분석(Intersection) = 안전하게 과소 추정

## Q6 -- Ch2 LICM 안전 조건 | 설명 제공

**문제**: LICM 적용 시 모든 invariant 문장을 무조건 preheader로 이동해도 안전한가? 안전하지 않은 경우와 안전 이동 조건을 서술하라.

**내 답변**: 모름 [--]

**정답 및 보완**: 무조건 이동하면 안전하지 않다. 루프가 0회 실행될 수 있는 경우, preheader로 옮긴 문장은 무조건 실행되어 원래 없던 부작용(예: divide-by-zero)이 발생할 수 있다. 안전 이동 조건: (1) 해당 문장이 루프의 모든 exit을 dominate해야 함 (루프 한 번이라도 돌면 반드시 실행되는 위치), (2) 또는 루프가 최소 1회 실행 보장 (do-while 변환), (3) 부작용 없는 문장이면 조건 완화 가능.

**핵심 개념**: LICM은 루프 0회 실행 시 원래 없던 부작용 발생 위험. dominate 조건 또는 최소 1회 실행 보장 필요

## Q7 -- Ch3 Live Variables 계산 | 부분 정답

**문제**: CFG에서 Live Variables의 USE, DEF를 구하고 backward 반복으로 IN/OUT 계산

**내 답변**: 직관적 답은 맞았으나 B2 USE 계산에서 c를 포함하는 오류 [X]

**정답 및 보완**: USE는 "해당 블록에서 정의되기 전에 사용되는 변수"만 포함. B2에서 c=a+b로 c를 먼저 정의한 뒤 d=c*a에서 c를 사용하므로 c는 exposed use가 아님. B2: USE={a,b}, DEF={c,d}. 마찬가지로 B3: USE={b,d}(a=d+1에서 d사용, c=a*b에서 b사용, a는 윗줄 정의), B4: USE={a,d}(b=d-1에서 d사용, d=a+b에서 a사용, 하지만 b는 윗줄 정의). 결과: B1 IN={}, B2 IN={a,b}, B5 IN={a,b}.

**핵심 개념**: USE(exposed use)는 블록 내에서 위->아래로 훑어 "아직 정의 안 된 변수의 사용"만 포함. 같은 블록 윗줄에서 정의된 변수는 USE에 넣지 않음

## Q8 -- Ch4 반복 수렴 조건 | 부분 정답

**문제**: 반복 알고리즘 수렴 보장을 위한 수학적 조건을 격자 높이, 전달함수 단조성과 연결하여 설명

**내 답변**: input 확장 시 f(input)도 확장, 변수/정의가 유한하므로 수렴, 아니면 무한 반복 [X] (단조성 방향은 맞으나 용어 부정확)

**정답 및 보완**: 수렴 3조건: (1) 유한 높이 격자 -- 값이 한 방향으로 변할 수 있는 횟수 제한, (2) 전달함수 단조성 -- x<=y이면 f(x)<=f(y), 입력이 커지면 출력도 같거나 커짐, (3) 합류 연산자가 격자의 meet/join과 일치. 단조적이지 않으면 값이 올라갔다 내려갔다 반복하는 진동(oscillation) 발생. 원인은 "유한하지 않아서"가 아니라 "단조성 위반으로 진동"이 정확한 표현.

**핵심 개념**: 수렴 = 유한 높이 격자 + 단조 전달함수 + 격자 합류 연산. 단조성 위반 시 무한 진동(oscillation)

## Q9 -- Ch2 Strength Reduction + IVE | 부분 정답

**문제**: 루프에 SR과 IVE를 순서대로 적용하여 변환 결과를 보여라

**내 답변**: SR로 shift 변환, IVE로 t1 삭제 [X] (SR의 핵심 변환 방식 오해)

**정답 및 보완**: SR은 곱셈을 shift로 바꾸는 것이 아니라, 루프 내 매 반복의 곱셈(i*4)을 preheader 초기화(t1=0) + 루프 내 덧셈(t1=t1+4)으로 대체하는 것. IVE는 원래 induction variable i를 제거하고 derived IV인 t1로 루프 종료 조건을 대체(i<100 -> t1<400). 최종 결과: t1=0; while(t1<400) { A[t1]=0; t1=t1+4; }

**추가 심화 (Q&A에서 도출)**: i*4를 i<<2로 바꾸는 것도 넓은 의미의 SR(곱셈->shift)이며 틀린 최적화는 아님. 하지만 수업에서 말하는 루프 SR의 핵심 가치는 곱셈/shift 자체를 덧셈으로 완전 대체 + IVE로 원래 IV(i) 자체를 제거하는 것. shift 방식은 i가 남아있어 IVE 적용이 불완전함. 시험에서는 수업 방식(곱셈->덧셈+IVE)으로 풀어야 함.

**핵심 개념**: SR = 곱셈을 preheader 초기화 + 루프 내 덧셈으로 대체. shift 변환도 SR의 일종이나, 수업 SR은 IVE와 결합하여 원래 IV 자체를 제거하는 것이 핵심

## Q10 -- Ch1 실행 방식 3가지 비교 | 부분 정답

**문제**: 순수 컴파일/인터프리터/하이브리드의 실행 시간 공식과 Java 하이브리드 이유

**내 답변**: 3가지 방식 구분 + Java 플랫폼 독립 이점 [X] (표현 정밀도 부족, 실행 시간 공식 미제시)

**정답 및 보완**: 순수 인터프리터는 사전에 타겟 기계어 파일(.exe 등)을 생성하지 않고, 실행 시점에 한 문장씩 해석하면서 인터프리터가 대응하는 기계어 동작을 수행하는 방식. 실행 시간 공식: 순수 컴파일 T=T_compile+T_native, 순수 인터프리터 T=T_interpret(사전 변환 없음), 하이브리드 T=T_compile_to_IR+T_interpret_IR(+T_JIT+T_native). Java 이점: 플랫폼 독립 + JIT으로 hot method만 선택적 기계어 컴파일하여 인터프리터의 이식성과 컴파일러의 실행 성능을 동시 확보.

**추가 심화 (Q&A에서 도출)**: "기계어 변환 없이 실행"이란 타겟 프로그램의 기계어 파일을 생성하지 않는다는 뜻이지, CPU가 기계어 없이 동작한다는 뜻이 아님. 인터프리터 자체가 기계어로 작성된 프로그램이며, 소스 코드를 해석한 결과로 현재 CPU에 맞는 기계어 동작을 수행시킴. 컴파일러와의 핵심 차이: 컴파일러=실행 전 기계어 파일 생성, 인터프리터=기계어 파일 미생성+실행 시점에 해석하며 수행.

**핵심 개념**: 컴파일러 vs 인터프리터의 구분은 "타겟 기계어 파일 사전 생성 여부". 인터프리터도 궁극적으로 CPU의 기계어를 통해 실행되지만, 타겟 프로그램의 기계어를 별도로 생성하지 않음
